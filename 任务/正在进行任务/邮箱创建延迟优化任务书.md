# 任务书：邮箱创建延迟优化

> 优先级：高 | 预计工作量：30 分钟 | 审核人：CTO

---

## 一、任务背景

当前 `codex-register` 项目中，注册流程会**提前创建邮箱**（coroabet 临时邮箱），不管后续注册是否成功。这导致失败的号码也浪费了邮箱资源。

**当前流程（浪费邮箱）：**
```
取号 → 创建邮箱 → 注册 OpenAI → 等验证码 → 提交验证码 → CPA 入库
        ↑
        这里就创建了，后面失败就浪费了
```

**目标流程（按需创建）：**
```
取号 → 注册 OpenAI → 等验证码 → 收到验证码后才创建邮箱 → 提交验证码 → CPA 入库
                                      ↑
                                      确定需要了才创建
```

---

## 二、技术分析

### 邮箱实际使用位置

邮箱只在 **CPA OAuth 入库阶段** 使用（`cpa-registration.ts` 的 Step 3），不在 OpenAI 注册阶段使用。

依赖关系：
- `authPhoneSignupHTTP()` — 注册 OpenAI，触发短信，等验证码 → **不需要邮箱**
- `completeAboutYou()` — 生成名字/生日（用 faker）→ **不需要邮箱**
- CPA OAuth（`authLoginViaCpaAuthorizeURL`）— 入库阶段 → **需要邮箱**

### 当前代码结构

**worker-scheduler.ts（调用方）：**
```
Step 1: 取号（HeroSMS）                    ← 行 ~130-155
Step 2: 创建邮箱（coroabet）                ← 行 ~157-183  ← 要删除这步
Step 3: 调用 runCpaRegistration(task)       ← 行 ~204
```

task 对象包含：
- `workerId`, `attemptId`, `phoneLease`, `phoneNumber`
- `bindEmail` — 邮箱地址                ← 要移除，改为内部创建
- `fetchAddEmailOtp` — 获取邮箱验证码的回调 ← 要移除，改为内部创建

**cpa-registration.ts（被调用方）：**
```
Step 1: 注册 OpenAI + 等验证码            ← 行 ~87-140
Step 2: CPA OAuth                         ← 行 ~142+   ← 邮箱在这里用
  ① 获取授权 URL
  ② OAuth 登录（需要 bindEmail）
  ③ 提交 callback 入库
  ④ 拉取 auth 文件
```

## 三、具体改动要求

### 文件 1：`src/worker-scheduler.ts`

**删除：** Step 2 整个邮箱准备块（约行 157-183）

**修改 task 对象：** 去掉 `bindEmail` 和 `fetchAddEmailOtp` 字段，改为可选或在 cpa-registration 内部创建。

当前代码（要改的部分）：
```typescript
// Step 2: 准备邮箱              ← 删除这整块
let bindEmail: string;
let fetchAddEmailOtp: () => Promise<string>;
try {
    const mailbox = await import("./mailbox.js");
    bindEmail = await mailbox.getEmailAddress();
    fetchAddEmailOtp = async () => {
        const startedAt = Date.now();
        console.log(`[scheduler] ${workerId} 等待邮件 OTP for ${bindEmail}`);
        return await mailbox.getEmailVerificationCode(bindEmail, { minTimestampMs: startedAt });
    };
    // ... db update ...
} catch (error) {
    throw new Error(`邮箱准备失败: ${(error as Error).message}`);
}
```

### 文件 2：`src/cpa-registration.ts`

**修改 `RegistrationTask` 接口：** `bindEmail` 和 `fetchAddEmailOtp` 改为可选字段。

**在 CPA OAuth 之前添加邮箱创建逻辑：** 在 `authPhoneSignupHTTP` 成功之后、CPA OAuth（`reportStatus("cpa_oauth")`）之前，添加邮箱创建代码。

插入位置：约行 141（`authPhoneSignupHTTP` 的 catch 块之后），行 142（`reportStatus("cpa_oauth")` 之前）。

要插入的逻辑：
```typescript
// 如果没有预分配邮箱，在收到验证码确认后才创建
let bindEmail = task.bindEmail;
let fetchAddEmailOtp = task.fetchAddEmailOtp;

if (!bindEmail) {
    const mailbox = await import("./mailbox.js");
    bindEmail = await mailbox.getEmailAddress();
    fetchAddEmailOtp = async () => {
        const startedAt = Date.now();
        console.log(`[cpa-registration] ${workerId} 等待邮件 OTP for ${bindEmail}`);
        return await mailbox.getEmailVerificationCode(bindEmail, { minTimestampMs: startedAt });
    };
    console.log(`[CPA] 邮箱已创建: ${bindEmail}`);
    
    // 更新 db
    db.updateWorkerSlot(workerId, { bind_email: bindEmail });
    db.updateAttempt(attemptId, { email: bindEmail });
}
```

注意：`cpa-registration.ts` 的 `runCpaRegistration` 函数需要接收 `db` 实例来更新记录。检查当前函数签名是否已经有 `db` 参数，如果没有需要从 task 中传入。

### 文件 3：`src/index.ts`

检查经典模式（`--codex-cpa`）中是否也有类似的提前创建邮箱逻辑，如果有，同样处理。

## 四、注意事项

1. **不要改变 `authPhoneSignupHTTP` 的逻辑** — 它不依赖邮箱，不需要动
2. **不要改变 CPA OAuth 的逻辑** — 它依赖邮箱，但邮箱来源改为内部创建
3. **保持 `fetchAddEmailOtp` 回调机制** — CPA OAuth 中通过这个回调获取邮箱验证码
4. **并发模式也要检查** — `concurrent-registration.ts` 中的邮箱创建逻辑是否也需要调整
5. **不要引入新依赖** — 只调整代码顺序
6. **构建验证** — 改完后必须 `npm run build` 通过

---

## 五、验收标准

1. `npm run build` 构建成功，无错误
2. 串行模式：`npm run dev -- --workflow codex-cpa-register --count 1 --skip-probe-trial` 正常运行

5. 并发模式同样行为正确
